# 승인 요청 동시성 제한 설정 근거

## 관측된 지연 시간

현재 테스트 그래프 기준으로 승인 API의 동기식 구간 지연 시간은 대략 다음 수준으로 보인다.

| 지표 | 관측값 |
| --- | ---: |
| 평균 latency | `20-50ms` |
| p95 | `90-110ms` |
| p99 | 보통 `100-200ms`, 순간적으로 `250-350ms` |

## 필요 동시성 추정

Little's Law 기준:

```text
필요 동시성 = RPS * latency
```

`200 RPS` 기준으로 계산하면 다음과 같다.

| 기준 | 계산 | 필요 동시성 |
| --- | ---: | ---: |
| 평균 `30ms` | `200 * 0.03` | `6` |
| p95 `100ms` | `200 * 0.10` | `20` |
| p99 `300ms` | `200 * 0.30` | `60` |

## 판단

현재 목표는 `256MiB` 힙에서 메모리를 늘리지 않고 안정적으로 버티는 것이다. 따라서 p99 순간 지연까지 모두 수용하려고 `max`를 `60` 이상으로 잡으면, 동시에 살아 있는 요청 객체가 많아져 GC 압박이 커질 수 있다.

가상 스레드는 스레드 자체 비용은 작지만, 요청이 처리되거나 대기하는 동안 잡고 있는 객체는 사라지지 않는다. 예를 들면 servlet request/response, Jackson DTO, MDC 값, 로그 이벤트, 트랜잭션 관련 객체 등이 힙에 남는다.

따라서 필터는 p95 수준의 동시성 요구량을 기준으로 잡고, 짧은 버스트만 제한적으로 대기시키는 방향이 적절하다. 그 이상은 빠르게 `429 Too Many Requests`로 밀어내는 것이 힙 보호에 유리하다.

## 추천 시작값

```yaml
payment:
  approval:
    concurrency:
      min: 0
      max: 24
      max-wait: 24
      max-wait-time: 100ms
```

## 설정 의미

| 설정 | 의미 | 근거 |
| --- | --- | --- |
| `max: 24` | 동시에 서비스 로직까지 진입 가능한 승인 요청 수 | p95 기준 필요 동시성 약 `20`에 약간의 여유를 둠 |
| `max-wait: 24` | active 슬롯이 꽉 찼을 때 대기 가능한 요청 수 | active + waiting 요청을 약 `48`개 수준으로 제한 |
| `max-wait-time: 100ms` | 대기 요청이 active 슬롯을 기다릴 최대 시간 | 관측된 p95 latency와 비슷한 수준. 오래된 버스트 요청을 빠르게 정리 |
| `min: 0` | Tomcat 설정 형태와 맞추기 위한 값 | 세마포어 필터에는 미리 유지할 플랫폼 스레드 개념이 없으므로 실질적 prewarm 의미는 없음 |

## 튜닝 프로파일

더 보수적인 설정:

```yaml
max: 16
max-wait: 16
max-wait-time: 50ms
```

추천 시작 설정:

```yaml
max: 24
max-wait: 24
max-wait-time: 100ms
```

힙이 안정적인데 `429`가 너무 많이 발생하는 경우:

```yaml
max: 32
max-wait: 32
max-wait-time: 150ms
```

## 다음 테스트 기준

먼저 아래 설정으로 테스트한다.

```yaml
payment:
  approval:
    concurrency:
      min: 0
      max: 24
      max-wait: 24
      max-wait-time: 100ms
```

테스트 결과에 따라 다음처럼 조정한다.

| 관측 신호 | 조정 방향 |
| --- | --- |
| `429`가 많고 힙이 안정적임 | `max`를 `32`까지 증가 |
| latency가 튀거나 GC 압박이 증가함 | `max`를 `16-20`으로 감소 |
| 짧은 버스트만 거절됨 | `max-wait`를 `32-48`까지 증가 |
| 힙이 계속 부족함 | `max-wait`를 `0-16`으로 감소 |

## 결론

현재 관측된 동기식 구간 성능 기준으로는 `max=24`, `max-wait=24`, `max-wait-time=100ms`가 가장 합리적인 시작점이다.

p95 기준 처리량은 대부분 수용하면서, p99성 순간 지연이나 과도한 버스트는 제한적으로 대기시키고 빠르게 거절할 수 있다.
